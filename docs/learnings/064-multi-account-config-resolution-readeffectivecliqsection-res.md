- **Multi-account config resolution (`readEffectiveCliqSection` + `resolveCliqConfig`).** The channel supports two config layouts: single-account (credentials at `channels.cliq.*` — the default, backward-compatible) and multi-account (per-account overrides under `channels.cliq.accounts.<id>.*` overlaid on the top-level section). Resolution rules: `accountId` null/undefined/`"default"` → top-level only; a non-default `accountId` with a matching `accounts.<id>` entry → that entry's fields shallow-merge over the top-level (per-account `allowFrom`/`selfSenderIds` REPLACE, not concatenate, matching the bundled-channel convention); a non-default `accountId` with NO matching entry → top-level fallback (a stray accountId never breaks resolution). `resolveCliqConfig` returns `accountId: null` for the single-account case so `CliqClientRegistry.buildKey` keys by `cc:<clientId>:<botId>`; per-account it returns the actual `accountId` so the registry keys by `acct:<id>` — this is what gives each bot its own OAuth token cache. `inspectCliqAccount` and `applyAccountConfig` follow the same convention: inspect reads the effective (merged) section; `applyAccountConfig` writes to the top-level for `"default"`/null and to `accounts.<accountId>` for any other id (so `openclaw configure` can provision a second bot without clobbering the first). The doctor's `dmAllowFromMode: "topOnly"` and the secret registry's channel-root-only paths remain correct for the single-account case; per-account secret paths (`channels.cliq.accounts.<id>.clientSecret`) are resolved at runtime by `resolveCliqSecretString` (the `path` arg is diagnostic-only — it does not affect value resolution, which reads `value` + `cfg.secrets`), but `openclaw secrets audit/apply` does NOT yet recognize per-account paths (the `cliqSecretTargetRegistryEntries` register only channel-root targets) — a multi-account operator must manage per-account secrets manually until that's extended.
