- **REST API v3 channel media posts are a v3 dead end.** v3 has NO byte-upload / multipart-attachment mechanism: the v3 Messages post endpoints (`POST /api/v3/channelsbyname/{name}/messages`, `POST /api/v3/chats/{chatId}/messages`, `POST /api/v3/bots/{bot}/messages`) take a JSON `{ text, reply_to?, sync_message? }` body with NO `attachments` field; v3 has no Files API at all (FILE_ID appears in the glossary but no endpoint exists — same finding as inbound file download); and the only v3 image surface is a Message-Card `images` slide that accepts PUBLIC HTTPS image URL strings only (no raw bytes) via the Message-Card channel endpoint `POST /api/v3/channels/{name}/message` (scope `ZohoCliq.Channels.CREATE`, user-context refresh-token grant — and it posts AS THE AUTHENTICATED USER, not the bot, same caveat as v3 channel cards). That path is strictly worse than the v2 multipart path (`POST /api/v2/channelsbyname/{name}/message?bot_unique_name={bot}` with a multipart `attachments` field — bot sender identity, raw bytes, any MIME type) for the plugin's `sendMedia` use case, so `CliqClient.sendMediaMessage` stays on `/api/v2/...` for BOTH DMs and channel posts REGARDLESS of the `apiVersion` opt-in, indefinitely. A regression test in `src/channel.test.ts` locks this invariant (channel media → `Channels.UPDATE` + refresh-token grant + v2 multipart; DM media → `Webhooks.CREATE` + `client_credentials` + v2 multipart — even when `apiVersion === "v3"`). Inline image rendering from a public URL via the v3 Message-Card `images` slide is a Phase 3 (rich messaging) concern, NOT a Phase 2 migration.
