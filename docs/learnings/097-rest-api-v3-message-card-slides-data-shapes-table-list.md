Per <https://www.zoho.com/cliq/help/restapi/v3/messagecards/> `slides` is a TOP-LEVEL array in the message payload (sits alongside `text` + `card`, NOT nested inside `card`) and is **theme-independent** — it composes with `modern-inline` / `prompt` / `poll` alike. Each entry is `{ type, title?, data }` where the `data` structure is per-type: `table` → `{ headers: string[], rows: Record<header,string>[] }` (each row object's KEYS map to a header — a row with extra keys silently drops them); `list` → `string[]`; `label` → `Array<{ label, value }>`; `images` → `string[]` of **HTTPS-only** URLs (non-HTTPS dropped by the renderer — Cliq requires publicly accessible HTTPS image URLs); `text` → `string`. The v3 docs do NOT document per-slide limits, so `normalizeV3Slide` applies defensive caps (headers ≤20, rows ≤50, list/label items ≤50, images ≤10, slides ≤20, cells ≤200 chars, text slide ≤4000 chars, slide title ≤100 chars) and silently drops invalid slides (empty headers / empty list / non-HTTPS images / empty text) so a malformed slide never fails the whole send — a degenerate `slides` array yields `undefined` (the payload omits `slides` entirely). The plugin's agent-facing input shape (`V3CardSlideInput` in `src/v3-card.ts`) uses field names that read naturally for an agent (`headers`/`rows` for table, `items` for list, `pairs` for label, `urls` for images, `text` for text); `readSlidesParam` in `src/message-actions.ts` parses the loose tool params into that shape defensively, and the renderer re-validates. The card's own multi-line `text` remainder is ALSO emitted as a `text` slide and comes BEFORE any input `slides` in the array, so a card with a body + a table slide emits `[{ type: "text", data: <remainder> }, { type: "table", ... }]`.
