- **Stop / abort intent: the SDK already cancels the in-flight run — a plugin channel only has to authorize the turn.** The runtime ships a fast-abort path (`tryFastAbortFromMessage`, exported internally from `abort-BI-_vvXG.js` / `openclaw/plugin-sdk/reply-runtime`) that `dispatchReplyFromConfig` invokes BEFORE agent dispatch (around `dispatch-DXwxohlw.js:1254`). It: (1) detects the abort via the shared `isAbortRequestText` (`ABORT_TRIGGERS` set: `stop`, `/stop`, `esc`, `halt`, `arrête`, `停止`, `стоп`, … — re-exported from `openclaw/plugin-sdk/reply-runtime`, so plugin channels use the SAME trigger set every channel uses); (2) authorizes via `resolveCommandSenderAuthorization({ ctx, cfg, commandAuthorized: ctx.CommandAuthorized })`; (3) on success cancels the active run (`acpManager.cancelSession` + `abortSessionRunTarget`), clears session followup/lane queues, stops spawned sub-agents, and replies with `formatAbortReplyText(stoppedSubagents)` (`"⚙️ Agent was aborted."`) via the channel's own `deliver` callback. The catch: the abort is authorized ONLY when `ctx.CommandAuthorized === true`. `finalizeInboundContext` derives `CommandAuthorized` from `CommandTurn.authorized`, which is `true` only when `CommandSource ∈ {"native","text"}` AND the input `CommandAuthorized` was `true` — a plain `"message"` source is forced to `authorized:false` (so just setting `CommandAuthorized:true` on a message turn does NOT survive finalization). A plugin channel that wants the SDK to abort on a stop intent must therefore set BOTH `CommandSource: "text"` AND `CommandAuthorized: true` on the `finalizeInboundContext` input for the abort turn (this mirrors the bundled Telegram channel, which gates the same fields via `resolveTelegramCommandIngressAuthorization` + `commandSource`). Marking a `stop` (no slash) turn as `source:"text"` is safe: `parseCommandName` returns `undefined` for non-slash bodies, so no command handler fires before the abort path (which runs first anyway). `resolveCommandSenderAuthorization` collapses to `commandAuthorized && isOwnerForCommands` for channels without a `commands.allowFrom`; Cliq ships none, `enforceOwnerForCommands` is off, and no owner allowlist is configured, so `isOwnerForCommands` is `true` and the gate is just `commandAuthorized`. `isAbortRequestText(text, { botUsername })` accepts an optional bot username to strip a `/stop@<bot>` slash-command suffix — pass `account.botName` for belt-and-suspenders when the caller may not have pre-stripped mentions. There is NO separate "cancel an in-flight run" hook on `runtime.channel` / `PluginRuntime` exposed to plugin channels — `cancelSession` lives on the ACP session manager (a gateway-internal surface), and `EmbeddedAgentQueueHandle.cancel("user_abort"|"restart"|"superseded")` is internal to the embedded-agent runner; the only plugin-reachable cancellation path is the fast-abort dispatch hook above. The bundled Telegram channel's OWN reply-fence abort controllers (`abortTelegramReplyFenceControllers`, line ~5918 of `bot-CBwSxkGA.js`) are an internal Telegram bot-module concern (it owns its dispatch loop) and are NOT available to plugin channels.
