- **`invoke.bot` button clicks re-enter as inbound messages — no Context handler needed for a self-contained confirm flow.** A v3 Message Card `invoke.bot` button (or the v2 bot-message `buttons[].action: "invoke"` equivalent) posts its `data.message` to the bot as an ordinary inbound message — the bot's Message handler receives it exactly like a typed message, with the clicking user as the sender. This means a plugin can build a self-contained confirm / multi-step button flow WITHOUT wiring a separate Cliq **Context handler** (the Deluge handler that fires on message interaction): the button's `data` carries a sentinel-prefixed payload, the existing Message-handler webhook recognizes the sentinel on the next call, and the dispatch path branches accordingly. The pattern: post a `prompt`-theme card with Confirm / Cancel `invoke.bot` buttons whose `data.message` is `__cliq_confirm__ <originalText>` / `__cliq_cancel__`; on the re-dispatch, strip the sentinel, mark the turn, and either skip the gate (confirm → dispatch the original text) or short-circuit with a canned reply (cancel → no agent turn). The original message text is encoded verbatim after the sentinel (cap ~1500 chars to stay well under Cliq's 5000-char message limit). The gate is a UX guardrail, not a security boundary — a user manually crafting the sentinel merely dispatches the text after it (exactly what they could have sent directly); the agent's own tool / permission policy still applies. Loop prevention is automatic: a confirm re-dispatch carries `confirmAction: "confirm"` so the gate is skipped.
