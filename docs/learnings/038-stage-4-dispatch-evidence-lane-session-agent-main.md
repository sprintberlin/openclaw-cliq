- **Stage-4 dispatch evidence: `lane=session:agent:main:cliq:`.** With a configured channel + default `ackPolicy` ("after_dispatch"), a valid Deluge POST returns `200 {"status":"received"}` only once `runtime.channel.inbound.run` resolves — so a 200 proves the full accept+dispatch path completed in the real gateway. The agent turn itself fails (no model configured: `ProviderAuthError`) and the outbound OAuth hop fails (fake credentials: `[cliq] oauth: response missing access_token`), but those failures are EXPECTED and are themselves the evidence the dispatch ran end-to-end. The authoritative "dispatched to an agent" marker in the gateway log is `[diagnostic] lane task error: lane=session:agent:main:cliq:<peerKind>:<peerId>` — the presence of `lane=session:agent:main:cliq:` means the inbound was routed to an agent session. Readiness must be probed with GET (→ 405 Method Not Allowed, route wired + channel configured); a 404 means the route is not registered yet. Do NOT probe readiness with the same POST payload used for the assertion — the dedupe claim would make the assertion probe return `200 "ok"` (skipped) instead of `200 {"status":"received"}` (dispatched). The gateway needs `--allow-unconfigured` (or `gateway.mode=local` in config) and `--auth none` (plugin routes with `auth:"plugin"` are not gateway-auth-gated, so `--auth none` is sufficient and avoids token plumbing).
