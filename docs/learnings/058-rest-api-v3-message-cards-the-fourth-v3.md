- **REST API v3 Message Cards (the fourth v3 migration family — `modern-inline` channel cards, issue #59).** v3 Message Cards replace the v2 bot-message `buttons` field (which the v3 message-post endpoints do NOT support) with a structured `card` object whose `theme` selects the rendering template. Per <https://www.zoho.com/cliq/help/restapi/v3/messagecards/> there are three themes: `poll` (voting options), `modern-inline` (header + labeled field sections + action buttons), and `prompt` (question + quick-reply buttons). The plugin implements `modern-inline` (issue #59) AND `prompt` (issue #63) for BOTH channel (non-DM) card posts (via the dedicated Message Card channel endpoint above) AND DM card posts (via the v3 bot-message endpoint's top-level `card` field — see below); `poll` remains unrendered. The v3 Message Card channel endpoint is `POST /api/v3/channels/{CHANNEL_UNIQUE_NAME}/message` — note THREE differences from the v3 channel *text* post: (1) the path is `/channels/{name}/message` (singular `message`), NOT `/channelsbyname/{name}/messages` (plural `messages`) — these are genuinely different endpoints; (2) the scope is `ZohoCliq.Channels.CREATE`, NOT `ZohoCliq.Webhooks.CREATE` (the text-post scope) and NOT `ZohoCliq.Channels.UPDATE` (the v2 channel-card scope) — a THIRD distinct channel scope; (3) the v3 Message Card docs do NOT document a `bot_unique_name` query param, so a v3 channel card posts **AS THE AUTHENTICATED USER** (the OAuth client owner), not as the bot — a behavior difference from the v2 channel card path; users who need bot sender identity for cards stay on `"v2"`. `ZohoCliq.Channels.CREATE` is a user-context scope the `client_credentials` grant cannot obtain a usable token for (same constraint as `Channels.UPDATE` — see issue #27), so the path routes through the refresh-token grant and throws when no `refreshToken` is configured (matching the v2 channel-card contract). The v3 `modern-inline` body shape: top-level `{ text?, card, slides? }` where `text` is the fallback (renders in notification previews / non-card clients), `card = { theme: "modern-inline", title (REQUIRED, ≤200 chars), thumbnail?, sections? (array of `{ title?, fields: [{title, value}] }`), buttons? (max 5, each `{ label (≤30 chars), action: { type, data } }`) }`, and `slides` (top-level, NOT nested in `card`) is an array of `{ type: "table"|"list"|"label"|"images"|"text", title?, data }` supporting-content blocks. Button action types: `open.url` (data.web), `invoke.function` (data.name), `system.api` (data.api), `invoke.bot` (data.bot_name + data.message — the closest v3 analog to v2's `action: "invoke"` + `data`: posts `message` back to the bot so the Deluge Message Handler receives it as an inbound message, same loop), `preview.url` (data.web). The plugin's v2→v3 button mapping (in `src/v3-card.ts`): `openurl`+url → `open.url` carrying `{ web: url }`; `invoke`+data → `invoke.bot` carrying `{ bot_name, message: data }` (requires the configured `botId` to address the bot — without it the button is dropped, since a bare `invoke.function` would need a registered Cliq Function name we don't have); `action: "api"` is unused by this plugin and dropped (no v3 `system.api` mapping — that calls a Cliq internal API path, not a generic HTTP hook). The v2 card's `text` (the agent reply combined with presentation body) is split into the v3 `title` (first line, ≤200 chars) + an optional `text` slide (the remainder); the full text is also kept as the top-level `text` fallback. v3 limits (vs v2): max 5 buttons per card (v2 allowed 10), title max 200 chars, button label max 30 chars. The 2xx response is `{ data: { id, card: {...} } }` (the existing `parseCliqMessageRef` already unwraps the v3 top-level `data` wrapper and reads `id`). The v3 Message Card DM endpoint `POST /api/v3/chats/{CHAT_ID}/messages` (scope `ZohoCliq.Messages.CREATE`) needs a chat id the DM send path does not have (it has a user id, not a `CT_xxx`), AND it posts AS THE AUTHENTICATED USER (not as the bot) — both disqualify it as the DM card path. Instead, DM cards route through the v3 **bot-message** endpoint `POST /api/v3/bots/{BOT_UNIQUE_NAME}/messages` with a top-level `card` field (issue #60): that endpoint accepts a `card` object directly, posts AS THE BOT (sender identity preserved — the bot unique name is in the URL path), addresses the recipient via `user_ids` (NO chat-id resolution needed), and uses the `ZohoCliq.Webhooks.CREATE` scope (`client_credentials`, NO refresh token required — unlike v3 channel cards which need the user-context `Channels.CREATE` scope). So the DM card v3 path is strictly cleaner than the dedicated Message Card chat endpoint and is the canonical v3 DM card send. The `poll` and `prompt` themes are not yet rendered (no agent-facing shape for them today).
