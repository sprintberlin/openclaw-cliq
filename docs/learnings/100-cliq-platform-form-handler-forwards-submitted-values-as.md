- **Cliq platform Form Handler (Phase 3 Forms).** Zoho Cliq's platform **Forms** (defined in the Cliq Bot editor under Edit Handlers → Form Handler) let a bot owner declare a structured form (text / number / dropdown / date / … fields). When a user submits the form, the bot's **Form Handler** Deluge script fires with the submitted field values in scope (each as a Deluge variable named after the field). The handler forwards them to a webhook via the same `invokeUrl` POST the Message / Mention / Welcome handlers use — there is **no separate REST CRUD for bot Forms** (the v3 REST API has bots / slash-commands / message-actions / schedulers CRUD but NOT bot Forms), so the form is defined in the Cliq console and the handler's `invokeUrl` is pointed at `/cliq/webhook`. No new OAuth scope: the Form Handler is a bot handler posting over the existing `x-cliq-webhook-secret`-authenticated transport (same as `Webhooks.CREATE`).
- **Payload shape is handler-defined (tolerant).** Cliq does not document a canonical forwarded payload for the Form Handler — the Deluge script builds the `Map()` it posts. The plugin's parser (`src/forms.ts`) therefore tolerates the submitted values under any of `values` / `form.values` / `form_data` / `formvalues` (including inside a `params` wrapper), the form name under `form.name` / `form.link_name` / `form_name`, and recognizes the payload as a form submission when `handler` is `"form"` (case-insensitive: `form` / `formhandler` / `form_handler`) OR a non-empty values object resolves. Field values may be primitives, arrays (multi-select), or Cliq `{ label, value }` dropdown objects (`normalizeCliqFormValue` coerces all three). A degenerate form with all-empty field values is dropped (no agent-readable content).
- **Routing: a form submission is a directed action at the bot.** The parser marks `isMention: true` for any recognized form submission, so a **group** form submission is admitted without a separate @mention (the user explicitly opened a form on the bot — the same logic as a `reply_to_bot` implicit mention). DM admission (`dmPolicy` / `allowFrom`) and self-message / dedupe guards apply unchanged. Form submissions also **bypass the `thinking.confirm` sensitive-keyword gate** (`src/confirm-gate.ts`): a structured submission is an explicit action, not free text to keyword-match, so a "reason: delete prod" field does not trip the gate. The synthesized body (e.g. `Form: approval_request\napprover: alice@…\npriority: High`) is the agent envelope's `Body`/`RawBody`; the raw structured map is surfaced on the inbound context as `FormValues` (string-keyed) + `FormName` so an agent tool can read it as structured data.
