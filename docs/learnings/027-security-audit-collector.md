- **Security-audit collector** (`openclaw security audit`). Distinct from the `doctor` adapter (which emits unstructured warning strings to `openclaw doctor`), the security audit emits structured `SecurityAuditFinding` objects (`{ checkId, severity: "info"|"warn"|"critical", title, detail, remediation? }`) to `openclaw security audit`. The collector is registered in `registerFull` via `api.registerSecurityAuditCollector(fn)` (a method on `OpenClawPluginApi`, NOT a `ChannelPlugin` adapter). The collector receives `OpenClawPluginSecurityAuditContext = { config, sourceConfig, env, stateDir, configPath }` and returns `SecurityAuditFinding[] | Promise<...>`. It MUST NEVER throw — a thrown collector aborts the whole audit sweep, so wrap the body in try/catch and degrade to `[]`. The shape mirrors the bundled Telegram collector (`security-audit-B4G860ds.js` → `collectTelegramSecurityAuditFindings`), which reads its `account.config` defensively and returns `[]` for an unconfigured channel. `SecurityAuditFinding` / `SecurityAuditSeverity` live in the internal `types-D7eu8baG.d.ts:5276`; a plugin channel defines a local structural `CliqSecurityAuditFinding` interface (structurally compatible) rather than importing the internal type. The checkId convention is `channels.<id>.<area>.<item>` (e.g. `channels.cliq.webhook_secret.missing`).
