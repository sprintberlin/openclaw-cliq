- **REST API v3 message delete (the third v3 migration family).** v3 Messages has NO single-message delete endpoint — only a bulk "Delete multiple messages" endpoint `DELETE /api/v3/chats/{CHAT_ID}/messagess?message_ids=<csv>` (the path's triple-s `messagess` is the published v3 path per the v3 Messages docs — it is NOT a typo in this code; the cURL examples on the docs page use the same triple-s consistently). A single-message delete is therefore a 1-element delete-multiple call (`message_ids=<oneId>`). The scope is `ZohoCliq.Messages.DELETE` — a user-context scope the `client_credentials` grant cannot obtain a usable token for (same constraint as `Messages.UPDATE`, `Channels.UPDATE`, `messageactions.CREATE` — see issue #27), so the path routes through the refresh-token grant (`resolveOutboundToken("ZohoCliq.Messages.DELETE", true)`) and throws when no `refreshToken` is configured — identical to the v2 delete path's contract (v2 reuses `Messages.UPDATE` via the refresh-token grant; the v2 path is left unchanged as the verified-live default). The v3 2xx response is a per-message result list `{ type: "message.delete_result", data: [{ id, status, error? }] }` where `status` is `"success"` or `"failed"`; for a 1-id delete the response carries exactly one entry, and success is `data[0].status === "success"` (parsed by `parseCliqDeleteResult`). A 2xx with no/empty/unmatched data is a logical failure → returns `false` (the caller — live-edit `safeDeleteDraft`, message-action `delete` — degrades gracefully and never breaks the turn); a non-2xx is classified + retried by `withSendRetry` (transient 429/5xx retried with backoff, 4xx fatal → throws `CliqSendError`), matching the v2 delete contract. The v3 delete-multiple accepts up to 25 ids per call (a 25-id cap enforced by the API with `message_delete_limit_exceeded`), and an admin-deleting-on-behalf path (`reason` + optional `comment` query params) exists but is unused by this plugin. Per the v3 OpenAPI / REST docs, v3 Messages has NO single-message edit or get/list-by-chat endpoint (only delete-multiple, post-in-channel/chat, forward, search) and v3 Chats has no message operations at all — so the v2 `editMessage` (`PUT /api/v2/chats/{chatId}/messages/{messageId}`) and `listChatMessages` (`GET /api/v2/chats/{chatId}/messages`) paths stay v2 indefinitely (a v3 dead end — confirmed, no swap available). `parseCliqMessageRef` already unwraps a top-level v3 `data` wrapper, but the delete-multiple `data` is an ARRAY (not a single object) so it needs its own parser (`parseCliqDeleteResult`); do not feed it through `parseCliqMessageRef`.
