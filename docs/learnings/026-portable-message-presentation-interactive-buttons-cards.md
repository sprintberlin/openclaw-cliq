- **Portable message presentation (interactive buttons/cards).** The SDK's portable interactive surface is `MessagePresentation` (`{ title?, tone?, blocks: [...] }`) with block types `text` / `context` / `divider` / `buttons` / `select`; buttons carry `{ label, action?: {type:"command"|"callback", ...}, value?, url?, style?, disabled? }`. Two distinct capability surfaces gate it, and they are NOT interchangeable:
  1. **`ChannelAgentPromptAdapter.messageToolCapabilities`** (agent-prompt) — only `inlineButtons` and `richText` are consumed, and BOTH are Telegram-specific (`inlineButtons` is a Telegram inline-keyboard *scope config*, `richText` is a Telegram Bot-API 10.1 rich-text term). A non-Telegram channel must return `[]`; advertising `inlineButtons` here does nothing useful for a plugin channel.
  2. **`ChannelMessageToolDiscovery.capabilities`** (message-action discovery, on `actions`) — the portable capability strings are `["presentation", "delivery-pin"]` (see `CHANNEL_MESSAGE_CAPABILITIES` in `types.core-DF7IXShG.d.ts:28`). Advertising `"presentation"` is the correct way to signal that the channel accepts interactive input via the shared `message` tool. Discord advertises `capabilities: ["presentation"]` and reads `ctx.params.components` in `prepareSendPayload` — params flow through to `handleAction`'s `ctx.params` regardless of whether a `schema` fragment declares them (the `schema` contribution only governs LLM discoverability/validation, not param passthrough).
  The `MessagePresentationButton`/`ChannelPresentationCapabilities` types are NOT re-exported from any public `openclaw/plugin-sdk/*` subpath — they live only in the internal `payload-BHJeg3MX.d.ts` / `outbound.types-B7fkjz65.d.ts`. A plugin channel defines local structural types matching the portable shapes (a `{ title?, tone?, blocks: PortableBlock[] }` is structurally compatible). (`MessagePresentation` itself, by contrast, IS re-exported from `openclaw/plugin-sdk/interactive-runtime`, as are `ChannelOutboundAdapter`/`ChannelOutboundPayloadContext` from `…/channel-runtime` and `ReplyPayload` from `…/core`.)
  See the dedicated **Outbound `renderPresentation` + `sendPayload` delivery path** Learning above for the full runtime flow (the card marker bridge, `hasReplyChannelData` gating, fallback semantics).
