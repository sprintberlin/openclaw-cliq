- **REST API v3 channel text post (the first v3 migration family).** `POST /api/v3/channelsbyname/{CHANNEL_UNIQUE_NAME}/messages?bot_unique_name={bot}` with body `{ text, reply_to?, sync_message? }` and scope `ZohoCliq.Webhooks.CREATE` (NOT `ZohoCliq.Channels.UPDATE` like the v2 channel endpoint). This is significant: `Webhooks.CREATE` is obtainable via the `client_credentials` grant (it's the same scope DMs use), so **v3 channel text posts do NOT require a user-context refresh token** — the v2 channel endpoint does (issue #27). The plugin exposes this as an opt-in `apiVersion: "v2" | "v3"` config (default `"v2"`) so the v2 path stays the verified-live default; per-account overrides apply via the shallow-merge resolution. v3 returns `204 No response` (HTTP 200 with body `{"Response Code":"204 No response"}`) — **no message id** — so live-edit streaming for channel posts degrades to block-streaming (the existing `resolveChannelChatId` + `listChatMessages` recovery path covers the missing id). v3 has **no `buttons` field** (buttons moved to Message Cards in v3 — see Phase 3), so `sendCard` stays on v2 regardless of `apiVersion`. The migration is deliberately one-endpoint-family-at-a-time per ROADMAP. The v3 path is built into `CliqClient.sendMessage` as a branch (`useV3Channel = !isDm && this.apiVersion === "v3"`); `resolveOutboundToken("ZohoCliq.Webhooks.CREATE", false)` is used for v3 channel (client_credentials, never the refresh-token grant even when one is configured — confirmed by the v3 docs and a unit test). The v3 Messages / Chats API has **no single-message edit / get / delete endpoint** (v3 Messages only: delete-multiple, post-in-channel, post-in-chat, forward, search; v3 Chats has no message operations at all) — so the v2 edit/delete/list/reaction paths may stay v2 indefinitely; confirm against the v3 OpenAPI spec before declaring that a dead end.
