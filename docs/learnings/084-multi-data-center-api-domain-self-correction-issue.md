- **Multi-data-center + `api_domain` self-correction (issue #46).** The plugin defaults to the EU endpoints (`https://accounts.zoho.eu` / `https://cliq.zoho.eu`) but the `oauthBase` / `apiBase` config overrides select any Zoho DC. The region‚Üíendpoints catalog lives in `src/region.ts` (`CLIQ_DATA_CENTERS`), consumed by the setup wizard's DC `select`, the doctor's DC validation, and the client. Zoho returns an `api_domain` field in BOTH token responses (`client_credentials` AND `refresh_token`) that encodes the account's region ‚Äî it is the *general* API host (e.g. `https://www.zohoapis.com`), NOT the Cliq host. The client parses it, maps the hostname back to the matching Cliq base (`https://cliq.zoho.<tld>` ‚Äî the raw `zohoapis` host is NEVER used as `apiBase` because it is not the Cliq host), and self-corrects `this.apiBase` + logs one warning when the detected region disagrees. The CA DC is special-cased: its `apiDomainHost` is `www.zohoapis.ca` but its `apiBase`/`oauthBase` are `cliq.zohocloud.ca` / `accounts.zohocloud.ca` (the `zohocloud.ca` TLD, not `zohoapis.ca`), so the catalog lookup ‚Äî not a TLD derivation ‚Äî is the source of truth. `oauthBase` is left unchanged on self-correction: a wrong `oauthBase` fails *before* any `api_domain` is returned (the first token request goes to `oauthBase`), so it cannot self-heal ‚Äî the setup-wizard DC prompt handles that at install time. Auth-failure bodies (`invalid_client` / `oauthtoken_scope_invalid` / generic auth 4xx) get a `verify your Zoho data center` hint appended (`appendCliqDataCenterHint` in `src/region.ts`, shared by the OAuth path in `client.ts` and `CliqSendError` in `send-retry.ts`) so a non-EU account hitting the EU endpoints surfaces a pointer instead of an opaque Zoho error.
