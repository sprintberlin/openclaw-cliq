- **`messaging` adapter (`ChannelMessagingAdapter`)** drives the runtime's session-binding / conversation-resolution pipeline. The type is exported from `openclaw/plugin-sdk/channel-runtime`; the helpers `buildChannelOutboundSessionRoute`, `buildThreadAwareOutboundSessionRoute`, `resolveThreadSessionKeys`, and the types `ChannelOutboundSessionRoute`/`ChannelOutboundSessionRouteParams`/`RoutePeer`/`RoutePeerKind`/`ChatType`/`OpenClawConfig` are re-exported from `openclaw/plugin-sdk/core` (NOT `channel-core`). The adapter lives on `base` (forwarded by `createChatChannelPlugin`'s `{ ...params.base, ... }` spread) — NOT on the top-level params. Key surfaces: `targetPrefixes` (provider prefixes, e.g. `["cliq"]`), `normalizeTarget(raw)` → canonical `cliq:<kind>:<id>` string, `resolveInboundConversation({from,to,conversationId,threadId,isGroup})` → `{conversationId, parentConversationId}`, `resolveDeliveryTarget({conversationId,parentConversationId})` → `{to, threadId?}`, `resolveSessionConversation({kind,rawId})` → `{id, threadId, baseConversationId, parentConversationCandidates}`, `resolveSessionTarget({kind,id,threadId})` → routable target string, `inferTargetChatType({to})` → `ChatType|undefined`, `resolveOutboundSessionRoute(params)` → `ChannelOutboundSessionRoute` (build via `buildChannelOutboundSessionRoute` so session-key orchestration stays in core), `formatTargetDisplay({target,display,kind})` → label, `targetResolver.{looksLikeId,hint,resolveTarget?}`. Cliq has no topics/threads, so `resolveSessionConversation` always returns `threadId:null` and `baseConversationId === id`. **Cliq user ids and channel unique names share no inherent discriminator** (unlike Telegram's signed chat ids), so the conversation id must carry a kind prefix (`user:<id>` / `channel:<name>`) to be self-describing; `resolveDeliveryTarget` rebuilds `cliq:<kind>:<id>` from that prefix. A bare `cliq:<id>` (no kind) is ambiguous and defaults to group (matching `normalizeCliqRouteTarget`); `parseCliqTarget` exposes an `explicit` flag so callers (`resolveInboundConversation`, `resolveOutboundSessionRoute`) can defer to an `isGroup` hint or a directory-resolved `kind` for bare ids instead of trusting the default. `resolveInboundConversation`'s `from` for DMs is `cliq:<senderId>` (bare, no kind prefix — matching Telegram's `telegram:<chatId>` convention), so it MUST consult `isGroup` to avoid misclassifying a DM sender id as a channel name.
