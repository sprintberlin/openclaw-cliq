- **Threading adapter (`ChannelThreadingAdapter`, type exported from `openclaw/plugin-sdk/channel-runtime`; `ReplyToMode` from `…/config-runtime`).** `createChatChannelPlugin`'s `threading` param accepts either a raw `ChannelThreadingAdapter` or a `ChatChannelThreadingOptions` shape. The three reply-mode option shapes (`core-CBhRRoge.d.ts:199`): (1) `{ topLevelReplyToMode: <channelId> }` — the string is a **channel id** fed to `createTopLevelChannelReplyToModeResolver`, which reads `cfg.channels?.[<channelId>]?.replyToMode ?? "off"` (so `topLevelReplyToMode: "reply"` is a BUG — it looks up `cfg.channels["reply"]` and resolves to `"off"`); (2) `{ scopedAccountReplyToMode: { resolveAccount, resolveReplyToMode, fallback } }`; (3) `{ resolveReplyToMode }` — direct function. Prefer form (3) for plugin channels. The adapter's `resolveReplyToMode` OVERRIDES the config-derived mode from `resolveConfiguredReplyToMode` (`reply-threading-J9AZHHNY.js:15`, which reads `channels.<id>.replyToModeByChatType[chatType]` → `channels.<id>.dm.replyToMode` (direct only) → `channels.<id>.replyToMode` → default `"all"`). Without an adapter, the default is `"all"`; with an adapter, the adapter controls. The mode filter (`createReplyToModeFilter`) strips `replyToId` from the reply payload for `"off"` (unless `allowExplicitReplyTagsWhenOff` + explicit tag), keeps it for `"all"`, and keeps only the first for `"first"`/`"batched"`. `resolveReplyTransport({ cfg, accountId, threadId, replyToId, replyToIsExplicit, replyDelivery })` → `{ replyToId, threadId }` consumed by `routeReply`/`delivery.runtime` to set the final `ctx.replyToId`/`ctx.threadId` on the outbound context (`route-reply-N1AhDxK6.js:106`, `delivery.runtime-Dg_mBtBZ.js:313`). Returning `threadId: null` clears it; omitting the key falls back to the input. `buildToolContext({ cfg, accountId, context, hasRepliedRef })` populates the shared `message` tool's `ChannelThreadingToolContext` (`currentChannelId`/`currentMessagingTarget`/`currentThreadTs`/`hasRepliedRef`) from the inbound `ChannelThreadingContext` (`To`/`ReplyToId`/…) — the MS Teams pattern (`channel-DKiirah7.js:956`) sets `currentChannelId: context.To`, `currentThreadTs: context.ReplyToId`. `resolveCurrentChannelId({ to, threadId })` is used by `resolveCurrentChannelTarget` for cron/delivery addressing — return `to` as-is when the channel has no thread-derived channel id (Cliq encodes the destination in `to` already). Cliq has no bot API reply-to / `parent_message_id` field, so replyToMode is honestly defaulted to `"off"` (carrying `replyToId` has no visible effect); `resolveReplyTransport` passes it through for logging and forces `threadId: null`.
