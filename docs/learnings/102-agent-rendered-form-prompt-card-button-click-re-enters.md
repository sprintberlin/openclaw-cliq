- **Agent-rendered form prompt-card button click re-enters as a structured `FormValues` entry on the inbound context (Phase 3, sub-part c).** An agent-rendered form (`message(action=send, form=…)`) posts each `select` field as a `prompt`-theme Message Card whose `invoke.bot` buttons carry `data: "__cliq_form__ <fieldName>=<value>"`. When the user taps a button, Cliq posts that string to the bot as an ordinary Message-handler message; the inbound parser (`parseCliqFormResponse` in `src/forms-render.ts`) recognizes the `__cliq_form__ ` prefix, splits the remainder on the first `=` (so a value may contain `=` or spaces), and surfaces it on the inbound context as `FormValues: { <fieldName>: <value> }` — the SAME surfacing the Cliq platform Form Handler path uses (`src/forms.ts`), so an agent tool can read the answer as structured params for a tool call rather than parsing free text. The agent envelope body is the clean `<fieldName>: <value>` rendering (sentinel stripped). A button click is a directed action at the bot → `isMention: true` (group responses admitted without a separate @mention) and the `thinking.confirm` gate is skipped (`parsed.formValues` short-circuits `isSensitiveInbound`). No `formName` on this path (the agent knows its own form).
- **Free-text replies are NOT structured.** The summary card (text / number fields, overflow select options) tells users to `reply with <name>: <value>` — those are plain text without a sentinel and re-enter as ordinary message text (`parsed.formValues` stays `undefined`). Only `__cliq_form__`-prefixed prompt-card button clicks are structured. This mirrors the broader `invoke.bot` sentinel pattern (learning 099) applied to the form-renderer's output, combined with the Form Handler's `FormValues` surfacing (learning 100).
