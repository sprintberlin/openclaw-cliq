- **Lifecycle `runStartupMaintenance` / `onAccountConfigChanged` / `onAccountRemoved` hooks.** `ChannelLifecycleAdapter` exposes these three hooks in addition to `detectLegacyStateMigrations` (`types.adapters-DptWV7FN.d.ts:526`). `runStartupMaintenance({ cfg, env?, log: {info?,warn?}, trigger?, logPrefix? })` is called once per loaded plugin at gateway start by `runChannelPluginStartupMaintenance` (`lifecycle-startup-CjAu-uvv.js:6`), wrapped in a try/catch that degrades a throw to a `warn` log — so it must do useful work but never block startup. `onAccountConfigChanged({ prevCfg, nextCfg, accountId, runtime })` is fired by the `applyChannelAccountConfig` write path (`channels-DU4qKSFa.js:347`) AFTER the new config is built but BEFORE it is committed; `onAccountRemoved({ prevCfg, accountId, runtime })` is fired by the account-removal path (`channels-DU4qKSFa.js:882`). There is NO "on stop"/shutdown hook — the adapter surface is start + config-change only, so "clean up on stop" cannot be taken literally; the closest equivalent is evicting stale per-account state on `onAccountRemoved`. For the Cliq channel these hooks (a) best-effort pre-warm the OAuth access token at startup so the first inbound after an idle gap skips the `accounts.zoho.eu` round-trip, (b) log the canonical webhook path (`/cliq/webhook`) operators must wire into the Deluge handler, and (c) evict the cached `CliqClient` for an account on config change/removal so a credential rotation (`clientSecret`/`botId`/`refreshToken`) is picked up immediately instead of reusing the stale cached client until process restart. Eviction must cover BOTH cache-key forms — `acct:<accountId>` (per-account layout) and `cc:<clientId>:<botId>` (single-account layout) — because a `clientSecret` rotation keeps the same `cc:` key, so the cached client holds stale credentials even though the key is unchanged; `CliqClientRegistry.getOrCreate` does not refresh an existing client's credentials.
