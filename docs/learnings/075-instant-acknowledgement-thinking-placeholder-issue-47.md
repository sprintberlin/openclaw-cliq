- **Instant acknowledgement / "thinking" placeholder (issue #47).** Since Cliq has no typing API, the substitute for a "typing…" cue is to post a lightweight placeholder message (e.g. `💭 …`) immediately on inbound, then EDIT it in place into the final agent reply — exactly one message, no duplicate. Editing a message needs the `ZohoCliq.Messages.UPDATE` scope, which `client_credentials` cannot obtain, so the placeholder flow is only usable when a `refreshToken` is configured; the gate is `thinking.mode === "placeholder" && !blockStreaming && account.refreshToken` (streaming preview's live-edit already shows progress, so the placeholder is a no-op there). The placeholder ref `{messageId, chatId?}` is threaded into `createLiveEditDeliver` as `initialDraft`; for group posts the send response carries only `{id}` (no chatId), so the chat id is resolved lazily via `CliqClient.resolveChannelChatId` on the first edit (cached per client — a `draftChatIdResolved` flag prevents redundant resolution calls when the channel is unresolvable). The "no stray `💭 …`" contract: when the edit cannot be performed cleanly (chat id unresolvable, edit API rejects, first-block overflow that can't be chunked into the draft) the placeholder is DELETED via `CliqClient.deleteMessage` and the reply is sent as a fresh message — a failed placeholder post/edit is swallowed + logged and never breaks or delays the turn. The no-reply case (the agent turn threw, or the dispatcher flushed zero blocks) is handled by wrapping `runtime.channel.inbound.run` in a `try/finally`: the `finally` checks `getLiveEditPlaceholderConsumed(deliver)` (exposed by `createLiveEditDeliver`, true iff a `deliver` call resolved the placeholder's fate) and, when the placeholder is still untouched, edits it into `thinking.failureText` (optional; e.g. `⚠️ No reply generated.`) or DELETES it when `failureText` is unset — so a stray `💭 …` never lingers. The `thinking` config (`{ mode: "off" | "placeholder", text, failureText? }`, default `"off"`) lives at `channels.cliq.thinking`, schema-validated in both the top-level and `channelConfigs.cliq` manifests with `uiHints`; `applyAccountConfig` writes it via `writeField("thinking")` so `openclaw configure` can provision it.
