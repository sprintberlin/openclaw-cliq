- **Instant acknowledgement / "thinking" placeholder (issue #47).** Since Cliq has no typing API, the substitute for a "typing…" cue is to post a lightweight placeholder message (e.g. `💭 …`) immediately on inbound, then EDIT it in place into the final agent reply — exactly one message, no duplicate. Editing a message needs the `ZohoCliq.Messages.UPDATE` scope, which `client_credentials` cannot obtain, so the placeholder flow is only usable when a `refreshToken` is configured; the gate is `thinking.mode === "placeholder" && !blockStreaming && account.refreshToken` (streaming preview's live-edit already shows progress, so the placeholder is a no-op there). The placeholder ref `{messageId, chatId?}` is threaded into `createLiveEditDeliver` as `initialDraft`; for group posts the send response carries only `{id}` (no chatId), so the chat id is resolved lazily via `CliqClient.resolveChannelChatId` on the first edit (cached per client — a `draftChatIdResolved` flag prevents redundant resolution calls when the channel is unresolvable). The "no stray `💭 …`" contract: when the edit cannot be performed cleanly (chat id unresolvable, edit API rejects, first-block overflow that can't be chunked into the draft) the placeholder is DELETED via `CliqClient.deleteMessage` and the reply is sent as a fresh message — a failed placeholder post/edit is swallowed + logged and never breaks or delays the turn. The agent-turn-failure case (no reply produced at all) leaves the placeholder in place — cleaning that up requires hooking the dispatch failure path, which is out of scope for v1 (the fuller thinking→generating→done status card is a Phase 3 item). The `thinking` config (`{ mode: "off" | "placeholder", text }`, default `"off"`) lives at `channels.cliq.thinking`, schema-validated in both the top-level and `channelConfigs.cliq` manifests with `uiHints`; `applyAccountConfig` writes it via `writeField("thinking")` so `openclaw configure` can provision it.
