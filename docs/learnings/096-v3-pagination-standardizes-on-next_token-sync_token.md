---
title: v3 pagination standardizes on next_token + sync_token
category: REST API v3
apis: [/api/v2/users,/api/v2/channels,/api/v3/datastores,/api/v3/chats,/api/v3/bots,/api/v3/slashcommands,/api/v3/messageactions,next_token,sync_token]
source: v3 Pagination docs
---
- **v3 standardizes ALL list endpoints on a two-token model:** `next_token` (paging — fetch the next page) and `sync_token` (incremental sync — fetch only records created/modified since the last request). v2 used six different pagination tokens (`next_token`, `sync_token`, `next_set_token`, `start_token`, `next_search_token`, `page_number`); v3 collapses these to just the two. The response body carries the cursor; pass it back verbatim as a `next_token` (or `sync_token`) query param on the next request — do not construct or modify it. **When `next_token` is absent from the response, the last page has been reached.** `limit` (page size) remains a request param; defaults/maxima vary by endpoint. The v3 response wraps the page under a top-level `data` array (e.g. `{ data: [...], next_token: "…" }`); v2 wraps under a resource key (`{ users: [...] }` / `{ channels: [...] }`) and MAY include a `next_token` (it was one of v2's six tokens). The plugin's `paginateList` helper (`src/pagination.ts`) follows `next_token` when present and falls back to `from`/`limit` offset pagination otherwise, so the v2 directory list calls (`listUsers` / `listChannels`) are forward-compatible with v3's `next_token` model even though they stay on the v2 paths (v3 has no org-directory equivalent — see learning 094). Ref: <https://www.zoho.com/cliq/help/restapi/v3/pagination/>.
