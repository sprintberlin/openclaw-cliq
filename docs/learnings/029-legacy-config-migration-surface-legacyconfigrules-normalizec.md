- **Legacy config migration surface (`legacyConfigRules` + `normalizeCompatibilityConfig` + `repairConfig`).** Two distinct doctor flows consume these: (1) `findLegacyConfigIssues` (`legacy-config-issues-CPAE70WU.js`) walks the raw config and for each rule emits `{ path, message }` when `getPathValue(root, rule.path) !== undefined && (!rule.match || rule.match(value, root))` — no `match` means "fire whenever the path is present". This is the `openclaw doctor` (no `--fix`) warning surface; `getBootstrapChannelPlugin(id)?.doctor?.legacyConfigRules` is the plugin-channel source. (2) `collectChannelDoctorCompatibilityMutations` calls `doctor.normalizeCompatibilityConfig({ cfg })` (in-memory compat pass) and `collectChannelDoctorRepairMutations` calls `doctor.repairConfig({ cfg, doctorFixCommand, env? })` (the `--fix` write path). Both return `ChannelDoctorConfigMutation = { config, changes, warnings? }` and short-circuit when `changes.length === 0`. For a pure/idempotent rewrite like snake_case→camelCase, `repairConfig` can delegate to `normalizeCompatibilityConfig` (sync) — there is no separate destructive step. The mutation MUST be non-destructive to its input (shallow-copy the section, mutate the copy, reassemble with spreads) because doctor may run the compat pass multiple times against the same `cfg` reference (idempotency check). `ChannelDoctorLegacyConfigRule` (`= LegacyConfigRule`) and `ChannelDoctorConfigMutation` are exported from `openclaw/plugin-sdk/channel-contract`. Bundled channels ALSO ship a separate `doctor-contract-api.{js,d.ts}` file exporting the same symbols (loaded by `doctor-contract-registry-BVJj1LXe.js`); plugin channels do NOT need that file — the `base.doctor` adapter is the canonical source for plugin channels.
