- **Inbound quote / reply context is a plugin-owned concern.** The Cliq message object the Deluge handler receives is the **new** message only — Cliq does NOT auto-attach the quoted/parent message's text to the bot (unlike Telegram's `message.reply_to_message`). The documented `message.reply_to` carries only the parent **message id** (string). To surface the parent *text* + *sender* the Deluge handler must fetch the parent and forward it as a sibling object (the parser tolerates `parent` / `quoted` / `parent_message` / `quoted_message` / `reply_to_message` at the payload root or under `message`, merging every source — `messageId` from `message.reply_to` + `text`/`sender` from a sibling object). When only the id is present, the plugin best-effort fetches the parent via `GET /api/v2/chats/{chatId}/messages` (matching by `messageId` in the recent-messages list — the chat-messages list endpoint, scope `ZohoCliq.Messages.READ`, requires the user-context refresh token, same `client_credentials` limitation as channel posts / edits). The quote text is prepended to the agent envelope `body` (the legacy `finalizeInboundContext` finalizer has no dedicated `ReplyToText` param consumed by the runtime — only `ReplyToId`/`ReplyToIdFull` are standard; the plugin appends `ReplyToMessageId`/`ReplyToText`/`ReplyToSenderId`/`ReplyToSenderName` for telemetry/correlation). A reply to the bot in a group sets the `reply_to_bot` / `quoted_bot` implicit-mention kinds (already wired in `resolveCliqMentionFacts` via the `opts` the dispatcher now passes) so a group reply to the bot is admitted without a fresh @mention.
