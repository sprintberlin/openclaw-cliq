title: Channel plugin HTTP routes are only registered when the channel is CONFIGURED

- **Channel plugin HTTP routes are only registered when the channel is CONFIGURED.** `shouldLoadChannelPluginInSetupRuntime` (`loader-channel-setup-RXs7a_z_.js:84`) returns true (→ "setup-runtime" registration mode, which does NOT call `registerFull`, so `api.registerHttpRoute` never runs) when none of the manifest's channels are present in `cfg.channels`. Only once `channels.<id>` is configured does the gateway load the plugin in "full" mode and wire the HTTP route. Consequences for the Stage-4 smoke: (1) `plugins inspect cliq --json --runtime` reports `httpRoutes: 0` on an unconfigured profile — that is expected and is NOT a regression (Stage-3 asserts load + capability, not routes); (2) the Stage-4 smoke MUST merge a `channels.cliq` block (with the schema-required `clientId`/`clientSecret`/`botId` plus `webhookSecret`/`dmPolicy`/`allowFrom`) into the profile config before starting the gateway, or the `/cliq/webhook` route is absent (HTTP 404, not 503). The plugin install writes `plugins.load.paths` + `plugins.entries.<id>.enabled` to `openclaw.json`; merge `channels.cliq` into that same file (a full overwrite is fine as long as the `plugins` block is preserved). `openclaw config set` rejects incremental channel-field writes because each intermediate state fails the schema's required-field validation — write the whole `channels.cliq` object at once.
