- **Outbound `renderPresentation` + `sendPayload` delivery path** (the reply-side presentation rendering). The runtime delivery flow (`deliver-DGFKMFsl.js`): when a `ReplyPayload` carries `presentation`, it calls `outbound.renderPresentation({ payload, presentation, ctx })` → expect a `ReplyPayload | null` back. The returned payload's `presentation` field is then stripped by the runtime (regardless of what you return). The runtime then re-checks `hasReplyPayloadContent({ presentation, interactive, channelData })`: if the rendered payload has non-empty `channelData` (any object with ≥1 key — `hasReplyChannelData` is just `Object.keys(value).length > 0`), it calls `outbound.sendPayload(ctx)`; otherwise it calls `sendText`/`sendFormattedText` as usual. So the bridge is: `renderPresentation` attaches a channel-native card marker to `payload.channelData` (e.g. `channelData.cliqCard = { buttons, text }`), and `sendPayload` reads that marker to route through the card send API, falling back to the plain send path when no marker is present. `presentationCapabilities` advertises what the channel can render (the runtime calls `adaptMessagePresentationForChannel({ presentation, capabilities })` to truncate/split blocks against `limits.actions` *before* calling `renderPresentation`, so the adapter receives already-truncated buttons). All three fields (`presentationCapabilities`, `renderPresentation`, `sendPayload`) live on `outbound.base` of `createChatChannelPlugin` (it `Omit`s only `sendText`/`sendMedia`/`sendPoll`, so they are forwarded onto the resolved `ChannelOutboundAdapter`). When `renderPresentation` returns `null`, the runtime falls back to `renderMessagePresentationFallbackText` (degraded plain text). `ReplyPayload`, `ChannelOutboundAdapter`, `ChannelOutboundPayloadContext`, and `MessagePresentation` are all exported from public subpaths (`openclaw/plugin-sdk/core`, `…/channel-runtime`, `…/interactive-runtime`).
