- **REST API v3 reactions / file-download / channel-chat-id-resolution are v3 DEAD ENDS (confirmed against the v3 OpenAPI / REST docs).** The v3 REST API has NO reactions endpoint anywhere — not under Messages (only delete-multiple, post-in-channel, post-in-chat, forward, search), not under Chats (no message operations at all), not under Threads, and not in the v3 sidebar (which exposes only Stars + Pin Messages as reaction-adjacent surfaces; the v2 `POST/DELETE /api/v2/chats/{chatId}/messages/{messageId}/reactions` with `{ emoji_code }` has no v3 analog). The add/remove reaction paths therefore stay on `/api/v2/...` indefinitely regardless of the `apiVersion` opt-in — locked by a regression test in `src/reactions.test.ts` so a future contributor cannot accidentally wire them to a v3 path that does not exist. The same docs pass confirms two more v3 dead ends the ROADMAP had flagged as "check the spec": (a) **file download** — v3 has no Files API (FILE_ID appears in the v3 glossary but no Files endpoint exists in the v3 sidebar / OpenAPI), so `GET /api/v2/files/{fileId}` stays v2; (b) **channel-chat-id resolution** — v3 chat retrieval is `GET /api/v3/chats/{CHAT_ID}` (by chat id, not by unique name) and `GET /api/v3/chats?type=channel` returns only chats the bot already converses with (a semantic change, not a clean swap), so the `GET /api/v2/channelsbyname/{name}` lookup stays v2. Together with the previously-confirmed message-edit / list-by-chat dead end, this means the ONLY v3-migratable outbound families are the three already migrated: channel text posts, bot DM posts, and message delete.
