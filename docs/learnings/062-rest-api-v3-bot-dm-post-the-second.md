- **REST API v3 bot DM post (the second v3 migration family).** The v3 analog of v2 `POST /api/v2/bots/{botId}/message` (with `userids`) is the v3 **Bots** API "Send a bot message" endpoint: `POST /api/v3/bots/{BOT_UNIQUE_NAME}/messages` — NOT `POST /api/v3/chats/{CHAT_ID}/messages`. This distinction is critical: the v3 "Post message in chat" (`/chats/{CHAT_ID}/messages`) posts **as the authenticated user** (the client owner), which would change the DM sender identity from "bot" to "user" and break the OpenClaw bot reply UX; the v3 "Send a bot message" (`/bots/{bot_unique_name}/messages`) posts **as the bot** (the bot unique name is in the URL path), preserving the v2 sender identity. The v3 bot-message body uses `user_ids` (a comma-separated string of ZUIDs/emails) — NOT v2's `userids` (one word, no underscore) — and supports `sync_message: true`, which makes the response carry `{ data: { message_id, chat_id } }` (the v3 wrapper; `parseCliqMessageRef` now unwraps a top-level `data` object so the same parser covers v2 + v3). Without `sync_message` the v3 bot-message response is `{ "Response Code": "204 No response" }` (no ids) — tolerated and degraded to block-streaming. The v3 docs list the endpoint's OAuth scope as `ZohoCliq.Webhooks.CREATE,ZohoCliq.BotMessages.CREATE` (comma-separated); the plugin requests ONLY `ZohoCliq.Webhooks.CREATE` (the one `client_credentials` can obtain and the existing v2 DM path already uses) — whether `BotMessages.CREATE` is additionally required is unconfirmed against a live org, so a fallback note in README §3c tells users to keep `apiVersion:"v2"` if their org requires the extra scope. The v3 bot-message endpoint also accepts `buttons` (max 5), `slides`, `card`, `bot` (override sender display name/avatar), `styles.markdown`, `reply_to`, `notify_edit`, `mark_as_read`, `thread_*` — richer than v2's bot-message; the current migration ports the plain-text DM shape (`{ text, user_ids, sync_message }`) AND the DM card shape (top-level `card` field — see the v3 Message Cards Learning; the renderer is `cliqCardToV3MessageCard` from `src/v3-card.ts`, the same renderer used for v3 channel cards). The v3 bot-message endpoint can ALSO post to a channel via a `CHANNEL_UNIQUE_NAME` query param (a unified send endpoint), but the channel path keeps using `/api/v3/channelsbyname/{name}/messages` for now (already shipped in the first increment) — unifying onto `/bots/{name}/messages` is a later cleanup. Scope/needsUserContext resolution needs NO change for v3 DMs: the existing calc (`scope = (isDm || useV3Channel) ? "Webhooks.CREATE" : "Channels.UPDATE"`, `needsUserContext = !isDm && !useV3Channel`) already routes any DM (v2 or v3) to `Webhooks.CREATE` + `client_credentials`; only the URL + payload differ (`useV3Dm = isDm && this.apiVersion === "v3"` branch).
