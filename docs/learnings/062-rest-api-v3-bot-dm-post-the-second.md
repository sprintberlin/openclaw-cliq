- **REST API v3 bot DM post (the second v3 migration family).** The v3 analog of v2 `POST /api/v2/bots/{botId}/message` (with `userids`) is the v3 **Bots** API "Send a bot message" endpoint: `POST /api/v3/bots/{BOT_UNIQUE_NAME}/messages` — NOT `POST /api/v3/chats/{CHAT_ID}/messages`. This distinction is critical: the v3 "Post message in chat" (`/chats/{CHAT_ID}/messages`) posts **as the authenticated user** (the client owner), which would change the DM sender identity from "bot" to "user" and break the OpenClaw bot reply UX; the v3 "Send a bot message" (`/bots/{bot_unique_name}/messages`) posts **as the bot** (the bot unique name is in the URL path), preserving the v2 sender identity. **Confirmed live (issue #85):** the v3 bot-message body uses the SAME recipient key as v2 — **`userids`** (one word, NO underscore) — a `user_ids` key is rejected with `400 {"code":"extra_key_found","message":"'user_ids' is an extra key in the JSON Object."}`. With `sync_message: true` the v3 bot-DM response carries `{ message_details: { "<userId>": { chat_id, message_id } } }` — NOT a `data.message_id` wrapper (other v3 endpoints DO wrap under `data`, so `parseCliqMessageRef` still unwraps a top-level `data` object when present). The `message_id` value is URL-encoded (e.g. `1783489987995%20224416859547` — `%20` = a space); `parseCliqMessageRef` decodes it once via `decodeURIComponent` so the v2 edit/delete URL builder (`PUT /api/v2/chats/{chatId}/messages/{messageId}`) encodes it exactly once via `encodeURIComponent` (no double-encode / stray `%2520`). Without `sync_message` the v3 bot-message response is `{ "Response Code": "204 No response" }` (no ids) — tolerated and degraded to block-streaming. The v3 docs list the endpoint's OAuth scope as `ZohoCliq.Webhooks.CREATE,ZohoCliq.BotMessages.CREATE` (comma-separated); the plugin requests ONLY `ZohoCliq.Webhooks.CREATE` (the one `client_credentials` can obtain and the existing v2 DM path already uses) and the endpoint accepts it on the verified-live org; a fallback note in README §3c / §4 tells users to set `apiVersion: { dmPost: "v2" }` if their org additionally requires the `BotMessages.CREATE` scope. The v3 bot-message endpoint also accepts `buttons` (max 5), `slides`, `card`, `bot` (override sender display name/avatar), `styles.markdown`, `reply_to`, `notify_edit`, `mark_as_read`, `thread_*` — richer than v2's bot-message; the current migration ports the plain-text DM shape (`{ text, userids, sync_message }`) AND the DM card shape (top-level `card` + `userids` + `sync_message`). Because v3 DM is strictly better than v2 (returns the message id v2 lacks), `dmPost` defaults to `"v3"`; the other migratable families (`channelPost`, `channelCard`, `delete`) stay `"v2"` by default (per-family `apiVersion` config, issue #85).
