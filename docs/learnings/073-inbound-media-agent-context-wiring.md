- **Inbound media → agent context wiring.** The plugin-channel inbound path (`runtime.channel.inbound.run` with `adapter.ingest`/`resolveTurn`) builds the `ctxPayload` itself via `runtime.channel.reply.finalizeInboundContext` (the SDK's `finalizeInboundContext` from `inbound-context-cqPd3Tht.js`, NOT the richer `finalizeChannelInboundContext`/`buildChannelInboundEventContext` the bundled channels use). That finalizer accepts the media fields DIRECTLY on the flat ctx object: `MediaPath` / `MediaUrl` / `MediaType` / `MediaPaths` / `MediaUrls` / `MediaTypes` (single-item legacy fields + plural arrays); it normalizes missing `MediaTypes` to a per-entry default. So a plugin channel attaches inbound media by downloading the file itself, writing it to a local temp path, and passing those flat keys into `finalizeInboundContext({...ctx, ...mediaFields})` — there is no `media` array param on this finalizer (the `media:` array param lives only on `finalizeChannelInboundContext`/`buildChannelInboundEventContext`, which the plugin `runtime.channel.reply` surface does NOT expose). The `ChannelTurnAdapter.preflight?.media` field (`InboundMediaFacts[]`) is consumed by `recordDroppedChannelTurnHistory` for dropped turns but is NOT merged into the ctx by the legacy `inbound.run` path — the plugin owns ctx media itself. Cliq file download endpoint is `GET /api/v2/files/{FILE_ID}` (scope `ZohoCliq.Attachments.READ` — <https://www.zoho.com/cliq/help/restapi/v2/files/#get-files>); the message object's `content.file.{id,name,type}` + `content.comment` caption is documented at <https://www.zoho.com/cliq/help/platform/cliq-objects/message-object.html>. Whether `client_credentials` can obtain a usable `Attachments.READ` token is undocumented; route the download through the refresh-token grant when configured (mirror channel posts/edits) and fall back to `client_credentials` otherwise — a failed download MUST degrade to "no media for that attachment", never break the turn. The runtime media-understanding pipeline reads from `MediaPath` (local file path); audio entries (`kind:"audio"`) are transcribed by the configured media-understanding provider, so set `transcribed:false` and let the runtime own transcription.
